# 통계 데이터 분석 결과 반영 수정 가이드

> 대상 노트북  
> - `recommend_score_prediction.ipynb`  
> - `sequential_route_recommendation.ipynb`  
>
> 목적: 두 노트북에 사용되는 방문자, 체류, 숙박, 소비, 내비게이션 등 **통계 데이터를 먼저 분석한 뒤**, 분석 결과에 따라 어떤 코드·가중치·설명 문서를 수정해야 하는지 정리한다.

---

## 1. 전체 수정 방향

현재 추천 시스템은 크게 두 단계로 나뉜다.

| 구분 | 노트북 | 역할 |
|---|---|---|
| 모델 2 | `recommend_score_prediction.ipynb` | 관광지 자체의 기본 추천 점수 생성 및 예측 |
| 모델 3 | `sequential_route_recommendation.ipynb` | 숙소/현재 위치에서 출발하여 조건별 다음 관광지 순차 추천 |

통계 데이터 분석 결과는 두 노트북에 다르게 반영해야 한다.

| 분석 결과 반영 위치 | 반영 방식 |
|---|---|
| `recommend_score_prediction.ipynb` | 관광지 기본 추천 점수 산식과 입력 변수에 반영 |
| `sequential_route_recommendation.ipynb` | 사용자 상황별 추천 점수 가중치, 동반 유형, 날씨, 거리 조건에 반영 |

즉, 모델 2는 **관광지 자체의 기본 매력도와 수요 지표**를 만드는 단계이고, 모델 3은 **현재 위치와 사용자 조건에 따라 추천 순서를 조정**하는 단계이다.

## 3. `recommend_score_prediction.ipynb` 수정 위치

`recommend_score_prediction.ipynb`는 통계 데이터를 이용해 `recommend_score`를 생성하고, 이 점수를 회귀 모델로 예측한다.  
따라서 통계 분석 결과가 바뀌면 **추천 점수 산식, 입력 변수, 가중치, 문서 해석**을 수정해야 한다.

---

### 3.1 외부 관광 수요 데이터 요약 변수 생성 부분

#### 수정 위치

노트북의 다음 섹션을 수정한다.

```text
4. 외부 관광 수요 데이터 요약 변수 생성
```

또는 코드상으로는 다음 변수를 생성하는 부분이다.

```python
avg_monthly_visitors
avg_visitor_growth_rate
avg_stay_time
avg_overnight_days
avg_overnight_ratio
avg_tourism_spending_outsider
avg_local_currency_spending
```

#### 수정해야 하는 이유

현재는 통계 데이터를 단순 평균값으로 요약하는 구조이다.  
하지만 분석 결과에 따라 단순 평균보다 최근값, 증가율, 표준화 점수, 외지인 비중 등을 반영하는 것이 더 적절할 수 있다.

#### 분석 후 수정 기준

| 분석 결과 | 수정 방향 |
|---|---|
| 최근 방문자 수가 증가 추세 | `avg_visitor_growth_rate` 또는 `regional_demand_score` 비중 증가 |
| 방문자 수는 많지만 감소 추세 | 방문자 수보다 증감률 반영 비중 확대 |
| 체류시간이 긴 지역 특성이 강함 | `avg_stay_time`을 추천 점수에 더 적극 반영 |
| 숙박방문자 비율이 높음 | 숙박형 관광지 또는 장거리 코스 추천 가중치 증가 |
| 관광소비액이 높음 | 소비 기반 관광 수요 점수 강화 |
| 지역화폐 소비가 높음 | 지역 상권·음식·쇼핑 관련 추천 점수 보정 |

#### 수정 예시

현재 구조가 단순 평균 중심이라면:

```python
avg_monthly_visitors = visitor_df['방문자수'].mean()
avg_visitor_growth_rate = visitor_df['방문자수증감률'].mean()
```

분석 결과 최근 3개월 흐름이 중요하다면 다음처럼 바꿀 수 있다.

```python
recent_visitor_df = visitor_df.sort_values('기준년월').tail(3)
avg_monthly_visitors = recent_visitor_df['방문자수'].mean()
avg_visitor_growth_rate = recent_visitor_df['방문자수증감률'].mean()
```

### 3.2 `regional_demand_score` 생성 부분

#### 수정 위치

```python
regional_values = pd.Series({
    'visitor': df['avg_monthly_visitors'].iloc[0],
    'growth': max(df['avg_visitor_growth_rate'].iloc[0], 0),
    'stay': df['avg_stay_time'].iloc[0],
    'overnight': df['avg_overnight_ratio'].iloc[0],
    'spending': df['avg_tourism_spending_outsider'].iloc[0],
    'local_currency': df['avg_local_currency_spending'].iloc[0],
})

df['regional_demand_score'] = regional_values.rank(pct=True).mean()
```

#### 수정해야 하는 이유

현재 `regional_demand_score`는 여러 관광 수요 지표를 단순 순위 평균으로 계산한다.  
하지만 분석 결과에 따라 방문자 수, 체류시간, 숙박, 소비 중 무엇을 더 중요하게 볼지 달라질 수 있다.

#### 수정 방향

| 분석 결과 | 수정 방향 |
|---|---|
| 방문자 규모가 가장 중요 | `visitor` 가중치 증가 |
| 최근 상승세가 중요 | `growth` 가중치 증가 |
| 체류형 관광이 중요 | `stay`, `overnight` 가중치 증가 |
| 소비 활성화가 중요 | `spending`, `local_currency` 가중치 증가 |

#### 수정 예시

가중 평균 방식으로 변경할 경우:

```python
regional_weights = {
    'visitor': 0.25,
    'growth': 0.15,
    'stay': 0.20,
    'overnight': 0.15,
    'spending': 0.15,
    'local_currency': 0.10,
}

regional_norm = regional_values.rank(pct=True)
df['regional_demand_score'] = sum(
    regional_norm[k] * w for k, w in regional_weights.items()
)
```

### 3.3 내비게이션 검색량 기반 `nav_score` 생성 부분

#### 수정 위치

```python
nav_df = read_csv_auto(paths['nav_volume'])
nav_avg = nav_df.groupby('목적지 유형', as_index=False)['목적지 검색량'].mean()
nav_map = dict(zip(nav_avg['목적지 유형'], nav_avg['nav_score_raw']))
```

그리고 다음 키워드 매핑 부분이다.

```python
NAV_KEYWORDS = {
    '음식':['음식','맛집','카페'],
    '쇼핑':['쇼핑','시장','마트'],
    '숙박':['숙박','호텔','펜션'],
    ...
}
```

#### 수정해야 하는 이유

내비게이션 검색량은 실제 관광객 관심도를 비교적 직접적으로 반영한다.  
따라서 분석 결과에서 특정 유형 검색량이 높으면 해당 유형의 `nav_score` 또는 추천 점수 가중치를 조정해야 한다.

#### 분석 후 수정 기준

| 분석 결과 | 수정 방향 |
|---|---|
| 음식 검색량이 압도적으로 높음 | 음식 관련 `nav_score` 보정 또는 음식 편중 방지 로직 추가 |
| 숙박 검색량이 높음 | 숙박 기반 코스 추천 강화 |
| 자연/관광지 검색량이 높음 | 자연관광·관광지 유형 가중치 증가 |
| 쇼핑 검색량이 낮음 | 쇼핑 추천 우선순위 낮춤 |
| 현지인과 외지인의 관심 유형 차이가 큼 | 사용자 유형별 `nav_score` 분리 고려 |

#### 수정 예시

음식 검색량이 너무 높아 추천 결과가 음식에 편중된다면:

```python
if row.get('target_tourism_type') == '음식':
    nav_score = nav_score * 0.85
```

또는 코스 다양성은 모델 3에서 처리하고, 모델 2에서는 원본 관심도를 그대로 유지할 수도 있다.

### 3.4 추천 점수 가중치 `WEIGHTS` 수정 부분

#### 수정 위치

```python
WEIGHTS = {
    'nav_score': 0.30,
    'type_quality_score': 0.20,
    'type_top1_prob': 0.15,
    'info_score': 0.10,
    'recent_score': 0.10,
    'regional_demand_score': 0.15,
}
```

#### 수정해야 하는 이유

통계 데이터 분석 결과는 최종적으로 `recommend_score` 산식의 가중치에 반영된다.  
예를 들어 내비게이션 검색량이 관광 수요를 가장 잘 설명한다면 `nav_score` 비중을 높이고, 체류·숙박 데이터가 핵심이면 `regional_demand_score` 비중을 높일 수 있다.

#### 수정 기준

| 분석 결과 | 수정할 가중치 |
|---|---|
| 내비게이션 검색량이 관광 관심도를 잘 설명 | `nav_score` 증가 |
| 관광 유형 확률이 추천 결과 품질에 중요 | `type_quality_score`, `type_top1_prob` 증가 |
| 데이터 품질 차이가 추천 결과에 크게 영향 | `info_score` 증가 |
| 최신 데이터 중심 추천이 필요 | `recent_score` 증가 |
| 방문자·숙박·소비 지표가 중요 | `regional_demand_score` 증가 |

#### 주의점

가중치 합은 항상 1.0이 되도록 조정해야 한다.

```python
sum(WEIGHTS.values()) == 1.0
```

### 3.5 모델 입력 변수 `numeric_features` 수정 부분

#### 수정 위치

```python
numeric_features = [
    'info_score',
    'recent_score',
    'nav_score',
    'type_quality_score',
    'type_top1_prob',
    'regional_demand_score',
    'avg_monthly_visitors',
    'avg_visitor_growth_rate',
    'avg_stay_time',
    'avg_overnight_days',
    'avg_overnight_ratio',
    'avg_tourism_spending_outsider',
    'avg_local_currency_spending',
    ...
]
```

#### 수정해야 하는 이유

통계 분석을 통해 새로 만든 파생 변수가 있다면 모델 입력 변수에 추가해야 한다.  
반대로 추천 점수 산식에 직접 사용된 변수가 그대로 입력에 들어가면 모델 성능이 지나치게 높아지는 **타겟 누수** 문제가 발생할 수 있다.

#### 수정 기준

| 상황 | 수정 방향 |
|---|---|
| 새 통계 파생 변수 생성 | `numeric_features`에 추가 |
| 추천 점수 산식 구성 요소를 그대로 예측에 사용 | 타겟 누수 여부 점검 |
| 실제 API 입력에서 받을 수 없는 변수 포함 | 제거 또는 별도 계산 로직 필요 |
| 지역 단위 상수값만 반복되는 변수 | 예측 모델에는 영향이 작으므로 해석용으로만 활용 가능 |

## 4. `sequential_route_recommendation.ipynb` 수정 위치

`sequential_route_recommendation.ipynb`는 모델 2의 기본 추천 점수를 바탕으로 현재 위치, 동반 유형, 선호 테마, 날씨, 미세먼지, 거리 조건을 반영하여 다음 관광지를 추천한다.  
따라서 통계 분석 결과는 **사용자 상황별 가중치와 거리 조건**에 주로 반영된다.

---

### 4.1 기본 추천 점수 컬럼 선택 부분

#### 수정 위치

```python
if "predicted_recommend_score" in df.columns:
    BASE_SCORE_COL = "predicted_recommend_score"
elif "recommend_score" in df.columns:
    BASE_SCORE_COL = "recommend_score"
```

#### 수정해야 하는 이유

모델 2에서 통계 분석 결과를 반영해 `predicted_recommend_score`를 새로 생성하면, 모델 3은 반드시 최신 결과 파일을 사용해야 한다.

#### 수정 기준

| 상황 | 수정 방향 |
|---|---|
| 모델 2를 다시 실행함 | 최신 `model2_recommend_score_dataset.csv` 사용 |

### 4.2 최종 점수 가중치 수정 부분

#### 수정 위치

노트북의 추천 점수 설명 섹션과 실제 계산 코드에서 다음 구조를 수정한다.

```text
final_score =
기본 추천 점수 × 0.30
+ 테마 적합도 × 0.20
+ 동반 유형 적합도 × 0.15
+ 날씨 적합도 × 0.15
+ 미세먼지 적합도 × 0.08
+ 거리 점수 × 0.09
+ 동반 인원 점수 × 0.03
```

#### 수정해야 하는 이유

통계 데이터 분석 결과에 따라 거리, 숙박, 체류, 동반 유형, 날씨 조건의 중요도가 달라질 수 있다.

#### 분석 후 수정 기준

| 분석 결과 | 수정 방향 |
|---|---|
| 가까운 거리 방문 비중이 높음 | `distance_score` 가중치 증가 |
| 장거리 외지인 방문 비중이 높음 | 거리 제한 완화, `distance_score` 가중치 감소 |
| 가족/단체 방문 비중이 높음 | `companion_score`, `group_size_score` 가중치 증가 |
| 체류시간이 길수록 관광 수요가 높음 | 기본 점수 또는 테마 적합도 비중 증가 |
| 날씨 영향을 많이 받는 관광 유형이 많음 | `weather_score` 가중치 증가 |
| 미세먼지 영향이 큰 실외 관광지 비중이 높음 | `fine_dust_score` 가중치 증가 |

#### 수정 예시

거리 영향이 중요하다고 판단되면:

```text
기본 추천 점수 × 0.28
+ 테마 적합도 × 0.20
+ 동반 유형 적합도 × 0.15
+ 날씨 적합도 × 0.13
+ 미세먼지 적합도 × 0.07
+ 거리 점수 × 0.14
+ 동반 인원 점수 × 0.03
```

### 4.3 거리 점수 및 거리 제한 수정 부분

#### 수정 위치

요청 스키마의 거리 제한 부분이다.

```python
"route": {
    "visited_contentids": [],
    "top_n": 5,
    "max_distance_km": 10
}
```

그리고 거리 점수를 계산하는 함수 또는 `recommend_next_places()` 내부의 거리 필터링 부분이다.

#### 수정해야 하는 이유

거리별 방문자 분포를 분석하면 실제 관광객이 어느 정도 거리까지 이동하는지 확인할 수 있다.  
이 결과를 기반으로 기본 거리 제한값과 거리 점수 감쇠 방식을 수정해야 한다.

#### 분석 후 수정 기준

| 분석 결과 | 수정 방향 |
|---|---|
| 5km 이내 방문 비중이 높음 | `max_distance_km` 기본값을 5~7km로 축소 |
| 10km 이내 방문 비중이 높음 | 현재 기본값 10km 유지 |
| 20km 이상 이동 비중이 높음 | `max_distance_km`를 20km 이상 또는 `None`으로 완화 |
| 숙박 관광객의 이동 거리가 김 | 숙박형 요청일 때 거리 제한 완화 |
| 당일치기/현지인 방문이 많음 | 거리 점수 가중치 증가 |

## 8. 최종 정리

통계 데이터 분석 결과는 단순히 보고서에 설명으로 추가하는 것이 아니라, 모델 2와 모델 3의 추천 로직에 직접 반영해야 한다.

`recommend_score_prediction.ipynb`에서는 방문자 수, 체류시간, 숙박 비율, 관광소비, 내비게이션 검색량 등을 바탕으로 `regional_demand_score`, `nav_score`, `recommend_score` 가중치를 수정한다.  
수정 후에는 모델을 다시 학습하고, MAE, RMSE, R²를 기준으로 최종 모델 성능을 다시 평가해야 한다.

`sequential_route_recommendation.ipynb`에서는 거리별 방문자 분포, 방문자 성·연령, 숙박 특성, 체류시간 분석 결과를 바탕으로 거리 제한, 동반 유형 점수, 동반 인원 점수, 테마 적합도, 최종 추천 점수 가중치를 수정한다.

결론적으로 수정 순서는 다음과 같다.

```text
1. 통계 데이터 분석
2. 모델 2의 수요 지표 및 추천 점수 산식 수정
3. 모델 2 재학습 및 결과 파일 재생성
4. 모델 3에서 최신 모델 2 결과 파일 사용
5. 모델 3의 거리·동반유형·테마·환경 가중치 수정
6. 추천 결과 확인
7. Markdown 정리 문서와 보고서 내용 수정
```
