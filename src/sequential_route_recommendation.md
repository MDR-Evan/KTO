# 숙소/현재 위치 기반 순차 관광지 추천 엔진 정리

> 파일명: `sequential_route_recommendation.ipynb`  
> 목적: `target_tourism_type.ipynb`의 관광지 유형 확률과 `recommend_score_prediction.ipynb`의 기본 추천 점수를 기반으로, 숙소 또는 현재 위치에서 다음에 방문할 관광지 Top 5를 순차적으로 추천한다.

---

## 1. 모델 개요

본 노트북은 **숙소/현재 위치 기반 순차 관광지 추천 엔진**을 구현한 것이다.  
이전 단계에서 생성된 관광지 유형 확률과 추천 점수 데이터를 활용하여, 사용자의 현재 위치와 여행 조건에 맞는 관광지를 추천한다.

모델 3의 핵심 목적은 단순히 인기 관광지를 추천하는 것이 아니라, 다음 조건을 함께 반영하여 **현재 위치 기준 다음 방문지 Top 5**를 제공하는 것이다.

| 반영 요소 | 설명 |
|---|---|
| 기본 추천 점수 | `recommend_score_prediction.ipynb`에서 생성한 `predicted_recommend_score` 또는 `recommend_score` |
| 현재 위치와의 거리 | 숙소 또는 선택 관광지 기준 거리 |
| 선호 테마 | 자연관광, 역사관광, 문화관광, 레저스포츠, 음식 등 |
| 동반 유형 | 혼자, 연인, 친구, 가족, 아이동반, 부모님, 단체 |
| 동반 인원 | 1명, 2명, 3~4명, 5명 이상 |
| 날씨 | 맑음, 흐림, 비, 눈, 폭염, 한파 |
| 미세먼지 | 좋음, 보통, 나쁨, 매우나쁨 |
| 방문 완료 목록 | 이미 방문한 관광지는 다음 추천에서 제외 |

본 노트북은 실제 API 서버를 구현하지 않고, API에서 들어올 요청 값을 `request_schema` 형태로 가정하여 추천 로직을 검증한다.

### 2.1 기본 입력 파일

모델 3은 모델 2의 결과 파일을 기본 데이터로 사용한다.

| 구분 | 내용 |
|---|---|

| 사용 목적 | 현재 위치 기반 순차 추천 후보 데이터 |

노트북에서는 실행 위치에 따라 다음 경로 후보를 자동으로 탐색한다.

```text
1. recommend_score_model_outputs/model2_recommend_score_dataset.csv
2. ../src/recommend_score_model_outputs/model2_recommend_score_dataset.csv
3. ../recommend_score_model_outputs/model2_recommend_score_dataset.csv
4. /mnt/data/recommend_score_model_outputs/model2_recommend_score_dataset.csv
```

### 2.2 필수 컬럼

`sequential_route_recommendation.ipynb`에서 최소한 필요한 컬럼은 다음과 같다.

| 컬럼 | 설명 |
|---|---|
| `contentid` | 관광지 고유 ID |
| `title` | 관광지명 |
| `mapx` | 관광지 경도 |
| `mapy` | 관광지 위도 |
| `predicted_recommend_score` | 모델 2에서 예측한 기본 추천 점수 |
| `recommend_score` | 산식 기반 추천 점수 |
| `type_prob_...` | 관광지 유형별 확률 |

추천 점수 컬럼은 다음 우선순위로 자동 선택한다.

```text
1순위: predicted_recommend_score
2순위: recommend_score
```

즉, `predicted_recommend_score`가 존재하면 이를 기본 추천 점수로 사용하고, 없을 경우 `recommend_score`를 사용한다.

## 3. 데이터 전처리

### 3.1 파일 로딩 보조 함수

### 3.2 기본 전처리 과정

입력 데이터 로딩 후 다음 전처리를 수행한다.

| 처리 항목 | 처리 내용 |
|---|---|

좌표가 없는 관광지는 현재 위치 기준 거리 계산이 불가능하기 때문에 추천 후보에서 제외한다.

## 4. API 입력값 구조

실제 서비스에서는 사용자의 숙소 또는 현재 위치, 동반 조건, 선호 테마, 환경 정보, 방문 완료 목록이 API 요청으로 들어온다고 가정한다.

### 4.1 요청 예시

```json
{
  "current_location": {
    "name": "경복대학교 남양주 캠퍼스",
    "mapx": 127.211247,
    "mapy": 37.734991
  },
  "companion": {
    "type": "혼자",
    "count": 1
  },
  "preference": {
    "themes": ["레저스포츠", "음식"]
  },
  "environment": {
    "weather": "흐림",
    "fine_dust": "나쁨"
  },
  "route": {
    "visited_contentids": [],
    "top_n": 5,
    "max_distance_km": 10
  }
}
```

---

### 4.2 입력값 설명

| 입력 항목 | 하위 항목 | 설명 |
|---|---|---|
| `current_location` | `name` | 현재 위치명 또는 숙소명 |
| `current_location` | `mapx`, `mapy` | 현재 위치의 경도, 위도 |
| `companion` | `type` | 동반 유형 |
| `companion` | `count` | 동반 인원 |
| `preference` | `themes` | 선호 관광 테마 목록 |
| `environment` | `weather` | 현재 날씨 |
| `environment` | `fine_dust` | 현재 미세먼지 상태 |
| `route` | `visited_contentids` | 이미 방문한 관광지 ID 목록 |
| `route` | `top_n` | 추천 개수 |
| `route` | `max_distance_km` | 최대 추천 거리 제한 |

`current_location`은 첫 추천에서는 숙소 좌표가 되고, 이후에는 사용자가 선택한 관광지 좌표로 갱신된다.

## 5. 보조 함수 구성

### 5.1 점수 정규화 함수

`minmax()` 함수는 기본 추천 점수를 0~1 범위로 정규화한다.

| 함수 | 역할 |
|---|---|
| `minmax()` | 수치형 값을 0~1 범위로 정규화 |

기본 추천 점수의 스케일이 다른 점수 요소와 다를 수 있기 때문에, 최종 점수 계산 전 정규화하여 사용한다.

---

### 5.2 거리 계산 함수

`haversine_km()` 함수는 현재 위치와 각 관광지 좌표 간의 거리를 km 단위로 계산한다.

| 함수 | 역할 |
|---|---|
| `haversine_km()` | 위도·경도 좌표 기반 거리 계산 |

거리 계산에는 지구 곡률을 고려하는 Haversine 공식을 사용한다.

---

### 5.3 유형 확률 컬럼 처리 함수

| 함수 | 역할 |
|---|---|
| `get_type_col()` | 테마명을 `type_prob_테마명` 컬럼명으로 변환 |
| `safe_col()` | 특정 컬럼이 없을 경우 기본값으로 대체 |

이를 통해 특정 관광 유형 확률 컬럼이 없는 경우에도 추천 로직이 중단되지 않도록 처리한다.

---

## 6. 추천 점수 구성 요소

최종 추천 점수는 여러 하위 점수를 가중합하여 계산한다.

| 점수 요소 | 설명 | 가중치 |
|---|---|---:|
| `base_score_norm` | 모델 2의 기본 추천 점수를 0~1로 정규화한 값 | 0.30 |
| `theme_match_score` | 사용자의 선호 테마와 관광지 유형 확률의 일치도 | 0.20 |
| `companion_score` | 동반 유형에 따른 관광지 유형 적합도 | 0.15 |
| `weather_score` | 날씨 조건에 따른 관광지 유형 적합도 | 0.15 |
| `fine_dust_score` | 미세먼지 상태에 따른 관광지 유형 적합도 | 0.08 |
| `distance_score` | 현재 위치와 관광지 간 거리 기반 점수 | 0.09 |
| `group_size_score` | 동반 인원에 따른 관광지 유형 보정 점수 | 0.03 |

### 6.1 최종 추천 점수 공식

```text
final_score =
  base_score_norm × 0.30
  + theme_match_score × 0.20
  + companion_score × 0.15
  + weather_score × 0.15
  + fine_dust_score × 0.08
  + distance_score × 0.09
  + group_size_score × 0.03
```

최종적으로 사용자에게 표시하기 위해 다음과 같이 100점 척도로 변환한다.

```text
final_score_100 = final_score × 100
```

## 7. 테마 적합도 계산

`compute_theme_match_score()` 함수는 사용자가 선택한 선호 테마와 관광지 유형 확률의 일치도를 계산한다.

예를 들어 사용자가 `레저스포츠`, `음식`을 선택하면 각 관광지의 `type_prob_레저스포츠`, `type_prob_음식` 값을 평균하여 테마 적합도를 계산한다.

### 7.1 테마 별칭 처리

사용자가 입력한 테마명이 모델의 관광 유형명과 완전히 일치하지 않을 수 있으므로 별칭 매핑을 사용한다.

| 입력 예시 | 매핑 유형 |
|---|---|
| 자연 | 자연관광 |
| 역사 | 역사관광 |
| 역사문화 | 역사관광 |
| 문화 | 문화관광 |
| 체험 | 문화관광 |
| 레저 | 레저스포츠 |
| 스포츠 | 레저스포츠 |
| 맛집 | 음식 |
| 카페 | 음식 |
| 식당 | 음식 |
| 숙소 | 숙박 |

선호 테마가 없거나 유효한 유형 확률 컬럼이 없는 경우에는 기본값 0.5를 사용한다.

## 8. 동반 유형별 적합도 계산

`compute_companion_score()` 함수는 동반 유형에 따라 적합한 관광지 유형을 가중합하여 계산한다.

| 동반 유형 | 주요 반영 유형 |
|---|---|
| 혼자 | 자연관광, 문화관광, 음식, 관광지 |
| 연인 | 자연관광, 음식, 문화관광, 관광지 |
| 친구 | 레저스포츠, 음식, 관광지, 문화관광 |
| 가족 | 자연관광, 역사관광, 문화관광, 관광지 |
| 아이동반 | 문화관광, 관광지, 자연관광, 음식 |
| 부모님 | 역사관광, 자연관광, 문화관광, 음식 |
| 단체 | 관광지, 음식, 레저스포츠, 문화관광 |

예를 들어 `친구` 유형은 활동성과 식사 수요를 고려하여 `레저스포츠`와 `음식` 유형에 높은 가중치를 부여한다.

> 현재 동반 유형별 가중치는 데이터 분석 기반이라기보다 초기 규칙 기반 설정이다.  
> 향후 실제 사용자 로그나 설문 데이터를 통해 보정할 필요가 있다.

## 9. 날씨 적합도 계산

`compute_weather_score()` 함수는 날씨 상태에 따라 관광지 유형의 적합도를 다르게 계산한다.

| 날씨 | 주요 반영 유형 |
|---|---|
| 맑음 | 자연관광, 레저스포츠, 관광지, 음식 |
| 흐림 | 문화관광, 역사관광, 음식, 관광지 |
| 비 | 음식, 쇼핑, 문화관광, 숙박 |
| 눈 | 음식, 쇼핑, 문화관광, 역사관광 |
| 폭염 | 음식, 쇼핑, 문화관광, 숙박 |
| 한파 | 음식, 쇼핑, 문화관광, 숙박 |

맑은 날에는 야외 활동과 관련된 자연관광, 레저스포츠 비중을 높게 반영하고, 비·폭염·한파와 같은 기상 조건에서는 실내형 관광지 또는 음식·쇼핑·숙박 관련 유형을 높게 반영한다.

---

## 10. 미세먼지 적합도 계산

`compute_fine_dust_score()` 함수는 미세먼지 상태에 따라 관광지 유형 적합도를 계산한다.

| 미세먼지 상태 | 주요 반영 유형 |
|---|---|
| 좋음 | 자연관광, 레저스포츠, 관광지, 음식 |
| 보통 | 관광지, 자연관광, 문화관광, 음식, 역사관광 |
| 나쁨 | 음식, 쇼핑, 문화관광, 숙박 |
| 매우나쁨 | 음식, 쇼핑, 문화관광, 숙박 |

미세먼지가 좋을 때는 야외 관광 유형의 비중을 높이고, 미세먼지가 나쁠 때는 실내 체류가 가능한 유형을 상대적으로 높게 반영한다.

---

## 11. 동반 인원 점수 계산

`compute_group_size_score()` 함수는 동반 인원 수에 따라 관광지 유형 적합도를 보정한다.

| 인원 조건 | 주요 반영 유형 |
|---|---|
| 1명 이하 | 음식, 문화관광, 자연관광, 관광지 |
| 2명 | 자연관광, 음식, 문화관광, 관광지 |
| 3~4명 | 관광지, 자연관광, 음식, 레저스포츠 |
| 5명 이상 | 관광지, 음식, 레저스포츠, 숙박 |

동반 인원 점수는 전체 최종 점수에서 0.03의 낮은 가중치로 사용되며, 추천 결과에 약한 보정 요소로 작동한다.

## 12. 현재 위치 기준 Top-N 추천 함수

`recommend_next_places()` 함수는 실제 API 엔드포인트에서 호출될 핵심 추천 로직이다.

### 12.1 처리 흐름

| 단계 | 처리 내용 |
|---|---|
| 1 | API 요청에서 현재 위치, 동반 정보, 선호 테마, 환경 정보, 방문 목록 추출 |
| 2 | 이미 방문한 관광지 제외 |
| 3 | 현재 위치와 각 관광지 간 거리 계산 |
| 4 | 최대 거리 제한이 있으면 해당 거리 이내 관광지만 필터링 |
| 5 | 기본 추천 점수 정규화 |
| 6 | 테마, 동반 유형, 날씨, 미세먼지, 거리, 인원 점수 계산 |
| 7 | 가중합으로 최종 추천 점수 계산 |
| 8 | 최종 점수 기준 정렬 후 Top-N 반환 |
| 9 | 추천 이유 문장 생성 |
| 10 | 필요 시 CSV 파일로 저장 |

---

### 12.2 방문 완료 관광지 제외

이미 방문한 관광지는 다음 추천에서 제외한다.

| 사용 컬럼 | 설명 |
|---|---|
| `visited_contentids` | 사용자가 이미 방문했거나 선택한 관광지 ID 목록 |
| `visited_flag` | 방문 여부를 0 또는 1로 표시 |

이를 통해 사용자가 같은 관광지를 반복 추천받지 않도록 처리한다.

---

### 12.3 거리 점수 계산

거리 점수는 현재 위치에서 가까울수록 높아지도록 계산한다.

```text
distance_score = 1 / (1 + current_distance_km)
```

| 거리 | 거리 점수 해석 |
|---:|---|
| 가까움 | 점수가 높음 |
| 멀어짐 | 점수가 낮아짐 |

거리 점수는 최종 점수에서 0.09의 가중치로 반영된다.

---

### 12.4 추천 결과 정렬 기준

최종 추천 결과는 다음 기준으로 정렬된다.

```text
1순위: final_score 높은 순
2순위: current_distance_km 가까운 순
```

즉, 추천 점수가 같거나 비슷할 경우 현재 위치에서 더 가까운 관광지가 우선 추천된다.

## 13. 추천 이유 생성

`make_recommend_reason()` 함수는 추천 결과에 대한 설명 문장을 생성한다.

추천 이유에는 다음 정보가 포함된다.

| 항목 | 예시 |
|---|---|
| 선호 테마 적합도 | `선호 테마(레저스포츠, 음식) 적합도 0.72` |
| 동반 유형 적합도 | `동반유형(혼자) 적합도 0.45` |
| 날씨 적합도 | `날씨(흐림) 적합도 0.51` |
| 미세먼지 적합도 | `미세먼지(나쁨) 적합도 0.58` |
| 거리 | `현재 위치에서 3.2km` |

이를 통해 추천 결과가 단순 점수로만 제시되지 않고, 사용자가 추천 이유를 이해할 수 있도록 한다.

## 19. 추천 시스템 내 역할

모델 3은 전체 관광 추천 시스템에서 최종 추천 엔진 역할을 수행한다.

| 단계 | 역할 |
|---|---|
| 모델 1 | 관광지 유형 확률 생성 |
| 모델 2 | 관광지 기본 추천 점수 예측 |
| 모델 3 | 현재 위치와 사용자 조건을 반영한 순차 관광지 추천 |

모델 1과 모델 2가 관광지 자체의 특성과 기본 추천 점수를 산출한다면, 모델 3은 사용자의 실제 여행 상황을 반영하여 다음 방문지를 결정한다.

---

## 20. 한계점

| 한계 | 설명 |
|---|---|
| 규칙 기반 점수 조합 | 최종 추천 점수가 학습 모델이 아니라 규칙 기반 가중합으로 계산됨 |
| 동반 유형 가중치의 주관성 | 동반 유형별 관광지 적합도 가중치가 실제 사용자 로그 기반이 아님 |
| 날씨·미세먼지 가중치 단순화 | 날씨와 미세먼지에 따른 관광 유형 적합도를 단순 규칙으로 처리함 |
| 실제 이동시간 미반영 | 직선거리 기반 Haversine 거리만 사용하며 도로 이동시간은 반영하지 않음 |
| 운영시간 미반영 | 관광지 영업시간, 휴무일, 입장 가능 시간 등이 반영되지 않음 |
| 체류시간 미반영 | 각 관광지에서 머무는 예상 시간이 코스 생성에 반영되지 않음 |
| 사용자 피드백 미반영 | 사용자의 클릭, 선택, 만족도, 재방문 여부 등이 학습되지 않음 |

---

## 21. 개선 방향

| 개선 방향 | 설명 |
|---|---|
| 도로 이동시간 API 연동 | 카카오맵, 네이버지도 등 경로 API를 활용해 실제 이동시간 반영 |
| 실시간 날씨 API 연동 | 현재 날씨, 강수 확률, 기온, 미세먼지 등을 API로 실시간 반영 |
| 영업시간 반영 | 관광지 운영시간, 휴무일, 입장 마감 시간을 고려한 추천 |
| 체류시간 기반 코스 최적화 | 예상 체류시간과 여행 가능 시간을 함께 고려 |
| 사용자 로그 기반 가중치 학습 | 선택률, 저장률, 방문률, 평점 데이터를 활용해 가중치 자동 학습 |
| 코스 다양성 제어 | 같은 유형의 관광지만 반복되지 않도록 다양성 제약 추가 |
| 숙소 복귀 조건 추가 | 마지막 코스에서 숙소로 복귀하는 거리 또는 시간 반영 |
| LLM 연계 | 자연어 요청을 API 입력 스키마로 변환하고 추천 이유를 자연어로 생성 |

## 22. 최종 정리

본 노트북은 모델 1의 관광지 유형 확률과 모델 2의 기본 추천 점수를 활용하여, 숙소 또는 현재 위치 기준으로 다음 방문할 관광지를 추천하는 순차 추천 엔진을 구현하였다.

추천 과정에서는 기본 추천 점수뿐 아니라 선호 테마, 동반 유형, 동반 인원, 날씨, 미세먼지, 현재 위치와의 거리, 방문 완료 목록을 함께 반영한다.  
이를 통해 사용자의 실제 여행 상황에 맞는 관광지 Top 5를 제공하고, 사용자가 특정 관광지를 선택하면 해당 위치를 기준으로 다음 관광지를 다시 추천할 수 있다.

최종 추천 점수는 다음 요소의 가중합으로 계산된다.

| 요소 | 가중치 |
|---|---:|
| 기본 추천 점수 | 0.30 |
| 선호 테마 적합도 | 0.20 |
| 동반 유형 적합도 | 0.15 |
| 날씨 적합도 | 0.15 |
| 미세먼지 적합도 | 0.08 |
| 거리 점수 | 0.09 |
| 동반 인원 점수 | 0.03 |

모델 3의 결과물은 숙소 기반 1차 추천, 선택 관광지 기반 2차 추천, 자동 다단계 코스 생성 결과로 저장된다.  
이 구조는 이후 API 서버와 연결하여 사용자가 숙소, 동반 유형, 선호 테마, 날씨, 미세먼지 정보를 입력하면 실시간으로 다음 관광지를 추천하는 서비스로 확장할 수 있다.

다만 현재 추천 로직은 규칙 기반 가중합 방식이므로, 향후 실제 사용자 로그와 이동시간 API, 영업시간, 체류시간 데이터를 결합하면 더 현실적인 개인화 여행 코스 추천 시스템으로 발전시킬 수 있다.
